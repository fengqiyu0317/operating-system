## 问题1：如何使得函数返回时能够跳转到调用该函数的下一条指令，即使该函数在代码中的多个位置被调用？

### 答案：

在 RISC-V 架构中，使用 **`jal`（Jump and Link）** 和 **`jalr`（Jump and Link Register）** 指令来实现这一功能。

**核心机制：**
- `jal rd, imm[20:1]`：在跳转之前，先将 `pc + 4`（即下一条指令的地址）保存到 `rd` 寄存器中
- `jalr rd, (imm[11:0])rs`：同样在跳转前保存返回地址

通常使用 `ra`（x1）寄存器来保存返回地址。当函数返回时，执行 `ret` 伪指令（实际被翻译为 `jalr x0, 0(ra)`），跳转到 `ra` 中保存的地址。

**工作原理：**
1. 调用函数时：`jal ra, function_label` → 将下一条指令地址保存到 `ra`，然后跳转
2. 函数返回时：`ret` → 跳转到 `ra` 中保存的地址

由于返回地址是在**每次函数调用时动态保存**的，所以同一个函数在代码的不同位置被调用时，`ra` 中保存的是各自调用点的下一条指令地址，从而实现正确的返回。

## 问题2：对于一个函数而言，保证它调用某个子函数之前，以及该子函数返回到它之后（某些）通用寄存器的值保持不变有何意义？

### 答案：

**意义在于保证程序的正确性和可预测性：**

1. **编译器独立编译的限制**：编译器是独立编译每个函数的，一个函数无法知道它所调用的子函数修改了哪些寄存器。

2. **保持执行上下文**：函数在调用子函数前后，可能正在使用某些寄存器存储：
   - 局部变量的值
   - 中间计算结果
   - 循环计数器
   - 其他重要的程序状态

3. **避免数据丢失**：如果子函数随意修改这些寄存器，调用者函数的执行状态会被破坏，导致程序逻辑错误。

4. **函数调用上下文（Function Call Context）**：需要在函数调用前后保持不变的寄存器集合。通过约定哪些寄存器需要保存、由谁保存，可以确保函数调用的正确嵌套。

## 问题3：调用者函数和被调用者函数如何合作保证调用子函数前后寄存器内容保持不变？调用者保存和被调用者保存寄存器的保存与恢复各自由谁负责？它们暂时被保存在什么位置？它们于何时被保存和恢复（如函数的开场白/退场白）？

### 答案：

### 寄存器分类：

| 类型 | 定义 | 负责保存/恢复 |
|------|------|---------------|
| **调用者保存（Caller-Saved）** | 被调用函数可能覆盖的寄存器，需要调用者保存 | 调用者函数 |
| **被调用者保存（Callee-Saved）** | 被调用函数可能覆盖的寄存器，需要被调用者保存 | 被调用者函数 |

### 保存位置：
**栈（Stack）** —— 通过 `sp`（栈指针）寄存器指向的内存区域

### 保存与恢复时机：

**调用者函数（针对调用者保存寄存器）：**
1. **保存时机**：在调用子函数之前（`jal`/`jalr` 指令之前）
2. **恢复时机**：在子函数返回之后（`jal`/`jalr` 指令之后）

**被调用者函数（针对被调用者保存寄存器）：**
1. **保存时机**：函数开头（**开场 Prologue**）—— 分配栈帧，保存寄存器
2. **恢复时机**：函数结尾（**结尾 Epilogue**）—— 恢复寄存器，回收栈帧，执行 `ret`

### 典型的开场/结尾代码示例：

```asm
# 开场（Prologue）
addi  sp, sp, -64    # 分配 64 字节栈帧
sd    ra, 56(sp)     # 保存返回地址
sd    s0, 48(sp)     # 保存帧指针
addi  s0, sp, 64     # 更新帧指针

# ... 函数主体 ...

# 结尾（Epilogue）
ld    ra, 56(sp)     # 恢复返回地址
ld    s0, 48(sp)     # 恢复帧指针
addi  sp, sp, 64     # 回收栈帧
ret                   # 返回
```

## 问题4：在 RISC-V 架构上，调用者保存和被调用者保存寄存器如何划分的？

### 答案：

### RISC-V 寄存器分类表：

| 寄存器组 | 别名 | 保存者 | 功能 |
|---------|------|--------|------|
| x0 | zero | - | 恒为零，不受函数调用影响 |
| x1 | ra | **被调用者保存** | 返回地址寄存器 |
| x2 | sp | **被调用者保存** | 栈指针 |
| x3 | gp | - | 全局指针，程序运行期间不变 |
| x4 | tp | - | 线程指针，程序运行期间不变 |
| x5~x7 | t0~t2 | **调用者保存** | 临时寄存器 |
| x8~x9 | s0~s1 (fp) | **被调用者保存** | s0 可作为帧指针 |
| x10~x17 | a0~a7 | **调用者保存** | 函数参数/返回值（a0, a1） |
| x18~x27 | s2~s11 | **被调用者保存** | 保存寄存器 |
| x28~x31 | t3~t6 | **调用者保存** | 临时寄存器 |

### 总结：

**调用者保存寄存器：**
- `t0~t6`（临时寄存器）：x5~x7, x28~x31
- `a0~a7`（参数/返回值寄存器）：x10~x17

**被调用者保存寄存器：**
- `ra`（返回地址）：x1
- `sp`（栈指针）：x2
- `s0~s11`（保存寄存器）：x8~x9, x18~x27（其中 s0 可作为 fp）

**不受影响：**
- `zero`（x0）：恒为零
- `gp`（x3）和 `tp`（x4）：程序运行期间不变

## 问题5：`sp` 和 `ra` 是调用者还是被调用者保存寄存器，为什么这样约定？

### 答案：

**两者都是被调用者保存寄存器。**

### 为什么 `ra` 是被调用者保存？

1. **函数可能调用其他函数**：如果被调用函数内部还要调用其他函数，它会再次使用 `jal`/`jalr` 指令，这会覆盖 `ra` 的值。

2. **保存时机**：被调用函数需要在**函数开头**保存自己的返回地址（`ra`），在**函数结尾**恢复它，然后才能执行 `ret` 返回。

3. **如果函数不调用其他函数**：编译器可以优化掉 `ra` 的保存/恢复，因为它不会被修改。

### 为什么 `sp` 是被调用者保存？

1. **栈帧管理**：每个函数都需要分配自己的栈帧（通过减小 `sp`），并在返回前回收栈帧（通过增加 `sp`）。

2. **调用链的完整性**：函数返回时，`sp` 必须恢复到调用前的值，这样调用者函数才能继续正确地使用栈。

3. **LIFO 特性**：栈的后进先出特性决定了最后分配的栈帧必须最先回收，这正是被调用者的责任。

## 问题6：如何使用寄存器传递函数调用的参数和返回值？如果寄存器数量不够用了，如何传递函数调用的参数？

### 答案：

### 使用寄存器传递参数和返回值：

**参数传递：**
- 使用 `a0~a7`（x10~x17）这 8 个寄存器传递参数
- 按顺序使用：第一个参数用 `a0`，第二个用 `a1`，以此类推

**返回值传递：**
- 使用 `a0` 和 `a1` 传递返回值
- 单个返回值用 `a0`
- 如果返回值超过一个寄存器能表示的范围，可以使用 `a0` 和 `a1` 组合

### 寄存器数量不够用时：

当函数参数超过 8 个时，使用**栈**来传递额外的参数：

1. **调用者**在调用函数前，将超过 8 个的参数压入栈中
2. **被调用者**通过栈指针 `sp` 加上适当的偏移量来访问这些参数
3. 调用完成后，调用者负责清理栈上的参数

### 示例（假设有 10 个参数）：

```asm
# 调用者代码
# 前 8 个参数在 a0~a7 中
# 第 9、10 个参数压栈
sd    t0, -8(sp)     # 第 10 个参数
sd    t1, -16(sp)    # 第 9 个参数
addi  sp, sp, -16    # 调整栈指针
call  function       # 调用函数
addi  sp, sp, 16     # 清理栈上的参数
```

### 栈帧中的参数布局：

```
高地址
┌─────────────────┐
│   第 N 个参数    │  ← 通过 sp + offset 访问
│   ...           │
│   第 9 个参数    │
├─────────────────┤
│   a7 (第8个参数) │
│   ...           │
│   a0 (第1个参数) │
└─────────────────┘
低地址
```

## 总结

本节知识清单涵盖了函数调用的核心机制：

1. **`jal`/`jalr` 指令**：通过保存返回地址实现函数调用的返回
2. **函数调用上下文**：保证寄存器在函数调用前后的一致性
3. **调用者/被调用者保存约定**：分工合作，提高效率
4. **RISC-V 寄存器分类**：明确的分工便于编译器优化
5. **栈的作用**：保存函数调用上下文、传递额外参数
6. **参数传递**：优先使用寄存器，不够时使用栈

这些机制共同构成了完整的函数调用规范（Calling Convention），是实现操作系统内核的基础知识。
